# Architecture

## Goal

本プロダクトの中心は、英語能力そのものより先に発生する「保持・統合負荷」を下げることです。そのため、システム設計でもコンテンツ配信より先に、以下を支える構造を重視します。

- 逐次処理を支える UI 制御
- チャンク分割と骨格抽出
- 音声の短区間再生
- 処理落ち点のイベント収集
- 個別設定による表示密度調整

## Logical Components

### 1. Flutter Client

責務:

- UI 表示
- 学習セッション制御
- 音声再生 / 録音
- 一部ローカルキャッシュ
- オフライン時の限定利用
- イベント計測送信

主要画面:

- 認証
- オンボーディング
- ホーム
- Chunk Reader
- Listening
- Speaking Builder
- Rescue
- Analytics
- Settings

### 2. Go API

責務:

- 認証認可
- ユーザープロファイル管理
- コンテンツ配信
- セッション管理
- イベントログ受付
- 分析結果返却
- Python worker の呼び出し制御

設計方針:

- REST API を基本とする
- OpenAPI を契約の唯一の真実源に近づける
- 重解析は同期 API から分離する
- ログ系の write path は単純化する

### 3. Python Worker

責務:

- 英文チャンク分割
- 骨格抽出
- 音声停止点候補推定
- 発話テキスト解析
- 難易度推定

設計方針:

- モデル差し替え可能な構成
- API 本体とは独立デプロイ
- 解析結果は再利用できるようキャッシュ可能にする

### 4. PostgreSQL

保持対象:

- ユーザー情報
- 学習コンテンツ
- チャンク結果
- 学習試行
- 分析スナップショット
- アプリイベント

### 5. Redis

用途:

- ジョブキュー
- 短期キャッシュ
- レート制限
- 一時セッション情報

### 6. Object Storage

用途:

- 音声素材
- ユーザー録音
- 将来の教材アセット

## High-Level Flow

### Chunk Reader

1. クライアントがコンテンツを取得
2. API が既存チャンクを返却
3. 未解析なら worker に解析依頼
4. クライアントはモード切替しながら閲覧
5. 各チャンクの閲覧イベントを記録

### Listening

1. クライアントが音声アセットを取得
2. API が停止点情報とメタデータを返却
3. クライアントが短区間再生
4. 継続 / 中断 / 再表示イベントを送信

### Speaking Builder

1. クライアントがテンプレートを取得
2. ユーザーが短文単位で録音
3. 録音メタデータと文字起こしを API に送信
4. worker が詰まり位置や発話長を分析
5. API が簡易フィードバックを返す

## MVP Boundaries

MVP では次をやりすぎないことを明示します。

- リアルタイム会話支援
- 高度な自由英作文添削
- 医療的判定に見える診断
- 複雑なマイクロサービス分割

## Suggested Service Layout

```text
apps/mobile
services/api
services/worker
infra/postgres
docs
```

## Engineering Priorities

1. イベントログを最初から入れる
2. Chunk Reader を最初の価値提供機能にする
3. 解析処理は API から分離する
4. UI は一画面一目的を守る
5. 設定で負荷密度を下げられるようにする
